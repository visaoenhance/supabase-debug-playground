# Supabase Validation Skill — v1

This document defines a validation standard for any agent interacting with Supabase.
This skill enforces validation discipline but does not determine or guarantee execution context safety. Environment verification remains the responsibility of the user.

Drop this into `.cursorrules`, `CLAUDE.md`, `.github/copilot-instructions.md`, or any agent context file.

**The rule:** before reporting any Supabase action as complete, run the appropriate validation command and confirm it passes. Do not ask the user whether to validate — validate automatically.

---

## Core principle

A Supabase action is not done when the code is written or when the command exits 0. It is done when the validation passes.

---

## Environment Scope & Safety

This skill defines a validation standard. It does not require a playground harness.

There are two usage modes:

### 1. Playground / Teaching Mode (this repository only)

The loop used in this repo:

> Reset → Break → Run → Fix → Run → Verify

- `reset` restores a known baseline
- `break` intentionally injects a bug
- This is safe only in a controlled local demo environment
- These steps are not required in real-world usage

This loop exists only to demonstrate failure modes.

### 2. Real-World Usage (staging / production / live projects)

In practice, the correct workflow is:

> Run → Diagnose → Fix → Run → Verify

- Do not run destructive `reset` or `break` operations in shared or production environments
- Never modify RLS, schema, or deploy functions without understanding environment scope
- If environment (local vs staging vs production) cannot be confirmed, assume it is shared and avoid destructive commands

The only universal requirement of this skill is:

**Do not report a Supabase action complete until validation passes.**

---

## Agent Safety & Destructive Commands

This skill does not authorize destructive operations.

If an action involves any of the following:
- `supabase db reset`
- Dropping tables or policies
- Replaying migrations
- Modifying RLS policies
- Deploying edge functions to production
- Using the `service_role` key
- Writing to a live database
- Any command that alters schema or data

Then:
- Do not execute the command automatically
- Present the command clearly in chat
- Explain what it will do
- Require explicit user confirmation before execution
- If interactive confirmation is not available (e.g., autonomous agent mode), output the command and explanation only — do not execute
- If environment cannot be confirmed as local or approved staging, assume it is shared and do not execute destructive commands

Destructive steps must be surfaced as review-required actions, not silently added to automated task execution.

**Safety overrides automation.**

## Methodology — Validation-First Supabase Discipline

This skill is not a checklist.
It is a behavioral contract.

When Supabase is involved, validation is mandatory — not optional.

**If Supabase is detected, you must include a "Validation" checklist item in your plan before writing any code.**

> **Playground Loop:** Reset → Break → Run → Fix → Run → Verify
> **Real-World Loop:** Run → Diagnose → Fix → Verify
> **Rule:** If you can't Verify, you can't Done.

---

### 0. Activation Rule — When This Skill Applies

This skill MUST activate automatically whenever Supabase is part of the task intent
or any Supabase signal appears in code, config, or commands — including:

- `supabase-js` is imported or used
- `supabase.rpc()` is called
- `.from(...).insert()`, `.update()`, or `.delete()` is used
- `CREATE FUNCTION`, `CREATE POLICY`, or `ALTER TABLE` appears in SQL
- `supabase functions deploy` is run
- `supabase gen types` is relevant to the task
- Any migration touches schema or RLS
- Any file under `supabase/functions/` is modified
- Any file under `supabase/migrations/` or `supabase/seed.sql` is modified
- Any `supabase db *` command is run (push/reset/diff)

**When any of the above occur:**
- Add a validation step to the task plan before execution begins
- Execute that validation step as part of the task before reporting completion. If execution is not possible in the current environment, explicitly state this and provide exact commands for the user to run.
- Do not ask the user whether to validate
- Do not make validation conditional on user instruction
- Before executing any destructive or environment-altering command (e.g., `supabase db reset`, migration replays, policy drops, deploy to production), confirm the environment is local or explicitly approved. If confirmation cannot be obtained, output the command for review instead of executing it.

---

### 1. Done ≠ Code Written

A Supabase task is not complete when code compiles or the command exits 0.
It is complete only when the validation step passes.

**Definition of done:**
1. Expected failure reproduced (if fixing an existing issue)
2. Fix applied
3. Validation executed
4. Validation produces a binary pass result

Never report completion without having executed step 4.

---

### 2. Reproduce Before Fixing

Before modifying any code or schema:
- Run the failing action
- Confirm the failure matches the expected symptom (error code, HTTP status, response shape)
- Capture the exact error
- If the task is building new functionality (not fixing a failure), establish a baseline run before applying changes

Do not patch blindly. If you cannot reproduce the failure, stop and re-evaluate
assumptions before proceeding — the environment may not be in the state you expect.

---

### 3. Verification Is a Hard Gate

A valid verification step must:
- Produce binary pass/fail — not output for the human to interpret
- Assert the response shape, not just the exit code
- Confirm role behavior for any RLS change
- Confirm row ID returned for any CRUD operation
- Confirm schema sync for any migration

**"Looks correct" is not a valid success signal.**
**"Exit code 0" alone is not a valid success signal.**

---

### 4. Multi-Layer Awareness

Supabase spans multiple independent layers:
- Client SDK (supabase-js)
- PostgREST (HTTP translation layer)
- PostgreSQL (database, RLS, functions)
- Edge runtime (Deno, env vars, error handling)
- Type generation (supabase/types.gen.ts snapshot)

A fix at one layer can silently break another. Code that compiles correctly can fail
at the network layer. A policy that works for one role can block another.

Always validate the full end-to-end round trip — not just the layer you changed.

---

### 5. Scenario Completeness Rule

If an action affects multiple states, roles, or conditions — all must be tested.

| Action | Required scenarios |
|---|---|
| RLS change | Unauthenticated anon blocked + authenticated anon allowed + service_role allowed |
| CRUD insert | Row returned with `id` present — not just no error |
| Schema migration | Column present in live DB + column present in `types.gen.ts` |
| Edge function | HTTP 200 + structured JSON body + `request_id` present |
| RPC call | Returns expected data shape or returns structured error with `code` + `message` captured (e.g., Postgres error code `42703`) |

Testing only the scenario you care about leaves the rest unverified.
Incomplete scenario coverage is incomplete validation.

---

### Automatic Task Injection Rule

When Supabase activity is detected in a task:

1. **Add** a validation step to the task plan before execution begins
2. **Execute** validation as part of the task — not as an afterthought
3. **Report** result only after validation passes
4. **If validation fails:**
   - Diagnose the failure using the appropriate diagnostic command
   - Apply a fix
   - Re-run validation
   - Only report complete after validation passes

The agent does not wait for user instruction to add these steps.
The agent does not skip these steps if the user does not mention validation.
The agent self-enforces this contract on every Supabase task.

**Minimum viable validation:** Run the minimum validation that proves the change, then expand only if the change affects multiple roles or states. Do not run every scenario on every task.

**Required plan shape when Supabase is detected:**
1. Reproduce (run current behavior — confirm failure if fixing, baseline if building)
2. Diagnose (use the pattern's diagnostic command)
3. Apply minimal fix
4. Verify (binary pass/fail — not eyeballing)
5. Report completion

## Pattern table

| Action type | Validation method | Passes when |
|---|---|---|
| Edge function (local) | HTTP POST → check status + response shape | HTTP 200 + `body.ok === true` + `request_id` present |
| Edge function (production) | HTTP POST against live URL → same checks | HTTP 200 + `body.ok === true` + `request_id` present |
| RPC create / modify | Call the function → check return shape | No error + response contains expected fields |
| Insert / CRUD | `.insert().select().throwOnError()` → check returned row | Non-null array with `id` present |
| RLS policy change | Test all three roles | Unauthed anon blocked + authed anon allowed + service_role allowed |
| Schema migration | Regenerate types → diff | `supabase gen types typescript --local` produces no diff against committed `types.gen.ts` |

## Pattern 1 — Edge function (local)

**Trigger:** after any change to a Supabase edge function running locally via `supabase functions serve`.

**Validation steps:**
1. POST to the function URL (default: `http://localhost:54321/functions/v1/<name>`)
2. Assert HTTP 200
3. Assert response body is valid JSON with `ok: true`
4. Assert `request_id` is present in the response body

**Fail signal:** HTTP 500, empty body, no `request_id`, or `Deno.env.get(...)` returning `undefined`.

**Diagnostic:** errors stream to the `supabase functions serve` terminal — read them directly.

**Docs:**
- [Environment Variables / Secrets](https://supabase.com/docs/guides/functions/secrets) — `Deno.env.get()` and local dev secrets
- [Error Handling](https://supabase.com/docs/guides/functions/error-handling) — structured try/catch and error responses
- [Logging](https://supabase.com/docs/guides/functions/logging) — reading the `supabase functions serve` output
- [Troubleshooting](https://supabase.com/docs/guides/functions/troubleshooting) — common edge function errors

**Do not report done until:** HTTP 200 + `ok: true` + `request_id` confirmed.

---

## Pattern 2 — Edge function (production)

**Trigger:** after `supabase functions deploy <name>` to a real Supabase project.

**Validation steps:**
1. POST to `$PROD_SUPABASE_URL/functions/v1/<name>` with `Authorization: Bearer $PROD_SUPABASE_ANON_KEY`
2. Assert HTTP 200
3. Assert response body is valid JSON with `ok: true`
4. Assert `request_id` is present in the response body

**Fail signal:** HTTP 500 or unstructured body.

**Diagnostic:** Prefer CLI logs if available in your environment; otherwise use the Supabase dashboard function logs:
`https://supabase.com/dashboard/project/<PROJECT_REF>/functions/<name>/logs`

**Docs:**
- [Deploy to Production](https://supabase.com/docs/guides/functions/deploy) — `supabase functions deploy` steps and CI/CD
- [Environment Variables / Secrets](https://supabase.com/docs/guides/functions/secrets) — production secrets management
- [Logging](https://supabase.com/docs/guides/functions/logging) — dashboard log access for production functions
- [Troubleshooting](https://supabase.com/docs/guides/functions/troubleshooting) — production error patterns

**Do not report done until:** HTTP 200 + `ok: true` + `request_id` confirmed against the production URL.

## Pattern 3 — RPC create / modify

**Trigger:** after any `CREATE OR REPLACE FUNCTION` or migration that modifies an RPC.

**Validation steps:**
1. Call the RPC via supabase-js: `supabase.rpc('<function_name>', { ...args })`
2. Assert `error` is null
3. Assert response data contains the expected fields (at minimum: `id` for insert-style RPCs)

**Fail signal:** `error.code` is present — read `error.code`, `error.message`, `error.hint` as a unit.

**Diagnostic:** if error code is `42703` (undefined column), run:
```sql
SELECT pg_get_functiondef('<schema>.<function_name>(<arg_types>)'::regprocedure);
```
This shows the exact SQL executing in the live database.

**Docs:**
- [Database Functions](https://supabase.com/docs/guides/database/functions) — creating and debugging PL/pgSQL functions
- [JavaScript RPC](https://supabase.com/docs/reference/javascript/rpc) — `supabase.rpc()` usage and error shape
- [PostgREST Error Codes](https://supabase.com/docs/guides/api/rest/postgrest-error-codes) — PostgreSQL error code taxonomy (42703 and others)

**Do not report done until:** RPC returns no error + response shape is correct.

## Pattern 4 — Insert / CRUD

**Trigger:** after any insert, update, or delete operation via supabase-js.

**Validation steps:**
1. Always chain `.select().throwOnError()` onto the operation
2. Assert returned `data` is a non-null array
3. Assert the array contains a row with `id`

**Why this matters:** `.insert()` without `.select()` sends `Prefer: return=minimal` — PostgREST returns 204 with empty body. supabase-js translates this to `{ data: null, error: null }`. This is not confirmation the row was saved.

**Correct pattern:**
```ts
const { data } = await supabase
  .from("table")
  .insert({ ...values })
  .select()
  .throwOnError();
// data is now a non-null array if the insert succeeded
```

**Docs:**
- [Managing Tables](https://supabase.com/docs/guides/database/tables) — table operations and PostgREST conventions
- [JavaScript Insert](https://supabase.com/docs/reference/javascript/insert) — `.insert()` behaviour and `.select()` chaining
- [PostgREST Error Codes](https://supabase.com/docs/guides/api/rest/postgrest-error-codes) — `return=minimal` vs `return=representation` and response shapes

**Do not report done until:** insert returns a non-null array containing a row with `id`.

## Pattern 5 — RLS policy change

**Trigger:** after any `CREATE POLICY`, `DROP POLICY`, `ALTER TABLE ... ENABLE ROW LEVEL SECURITY`, or migration that touches RLS.

**Validation steps — all three must pass:**

1. **Unauthenticated anon** — attempt the restricted operation with a plain anon key and no auth session → assert blocked (error `42501`)
2. **Authenticated user** — attempt the same operation with an anon key + valid JWT session → assert allowed
3. **service_role** — attempt the same operation with the service_role key → assert allowed (service_role always bypasses RLS)

**Why three scenarios:** testing only the happy path misses the other two. A policy that blocks service_role is broken. A policy that allows unauthenticated users may be a security hole.

**Diagnostic commands:**
```sql
-- See all policies on a table
SELECT policyname, cmd, qual, with_check
FROM pg_policies WHERE tablename = '<table>';

-- Confirm RLS is enabled
SELECT relname, relrowsecurity FROM pg_class WHERE relname = '<table>';
```

**Key insight:** `service_role` has the `BYPASSRLS` attribute — it skips policy evaluation entirely. If something works from the Supabase dashboard but fails in the app, check whether an INSERT policy exists for the role your app uses.

**Docs:**
- [Row Level Security](https://supabase.com/docs/guides/database/postgres/row-level-security) — policies, roles, and `service_role` bypass behaviour
- [Debugging and Monitoring](https://supabase.com/docs/guides/database/inspect) — `pg_policies` and live schema inspection
- [Hardening the Data API](https://supabase.com/docs/guides/database/hardening-data-api) — securing public vs private schemas

**Do not report done until:** all three scenarios produce the expected result.

## Pattern 6 — Schema migration / type drift

**Trigger:** after any migration that adds, removes, or renames a column.

**Validation steps:**
1. Run: `supabase gen types typescript --local > supabase/types.gen.ts`
2. Run: `git diff supabase/types.gen.ts`
3. If diff is non-empty → expected (new column added). Commit the updated file.
4. If diff is empty and you added a column → something is wrong, check the migration ran.

**Drift detection:**
```sql
-- Confirm column exists in live DB
SELECT column_name, data_type, is_nullable
FROM information_schema.columns
WHERE table_schema = 'public' AND table_name = '<table>'
ORDER BY ordinal_position;
```

**Why this matters:** TypeScript compiles cleanly against stale types. There is no compile error when a column exists in the DB but not in `types.gen.ts` — the column is simply invisible to your code.

**Where this belongs in CI:** run `supabase gen types typescript --local > supabase/types.gen.ts && git diff --exit-code supabase/types.gen.ts` in CI after any deploy that includes a migration. Fail the build if the diff is non-empty.

**Docs:**
- [Generating TypeScript Types](https://supabase.com/docs/guides/api/rest/generating-types) — `supabase gen types` CLI usage and CI setup
- [Debugging and Monitoring](https://supabase.com/docs/guides/database/inspect) — `information_schema` queries for live DB inspection
- [Local Development](https://supabase.com/docs/guides/cli/local-development) — managing migrations and local/remote schema sync

**Do not report done until:** `types.gen.ts` reflects the current live schema and the file is committed.

---

## Constraints (apply to all patterns)

- Never skip verification because the code looks correct — run the check
- Never report a Supabase action complete based on exit code alone — assert the output shape
- If a verify step fails, diagnose before retrying — don't loop without reading the error
- Always reset to known-good state after a break/fix cycle only in controlled playground environments
