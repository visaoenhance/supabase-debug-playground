# EP10 — Query Performance & Indexing

**Pattern introduced:** EXPLAIN plan validation — assert that a date-range query uses an Index Scan (not Seq Scan) after adding the appropriate index

---

## What this episode covers

A receipts query that was fast in development becomes slow as the table grows. The query filters by `created_at` and returns the latest 25 rows, but without an index on `created_at`, PostgreSQL must scan every row in the table to find them. We run `EXPLAIN ANALYZE` to observe the Seq Scan, add a `created_at DESC` index, and confirm the planner switches to an Index Scan — reducing execution time proportionally to table size.

---

## What the viewer learns

- How to read a basic `EXPLAIN ANALYZE` output to identify Seq Scan vs Index Scan
- Why a date-range query on a large table without an index degrades linearly with row count
- How to create an index on `(created_at DESC)` to match the query's ORDER BY direction
- Why `ANALYZE` must be run after structural changes for the planner to pick up updated statistics
- The production consideration: `CREATE INDEX CONCURRENTLY` avoids holding an exclusive table lock

 Limit  (cost=...) (rows=25...)

**Say:** "Seq Scan on 10,000 rows — the database is reading every single row just to find the 25 newest. At 10k rows this might be 50ms. At 100k rows it could be 500ms. At 1M rows it's measuring in seconds. The cost grows linearly with the table."

### 3 · Diagnose

```bash
docker exec supabase_db_supabase-debug-playground psql -U postgres -c \
  "SELECT indexname, indexdef FROM pg_indexes
   WHERE tablename = 'receipts';"
```

**Expected:** No row for `receipts_created_at_idx`.

**Say:** "The index listing confirms it. We have a primary key index and a user_id index — but nothing on `created_at`. This query filters on `created_at` and orders by it. Without an index, the planner has no choice but to scan everything."

---

### 4 · Fix

```bash
pnpm ep10:fix
```

**Say:** "`ep10:fix` creates `receipts_created_at_idx ON receipts (created_at DESC)`. The `DESC` matches the query's ORDER BY direction — the planner can read the index in order and stop after 25 rows without a sort step. Then ANALYZE refreshes the statistics."

Production note:
```sql
-- In production, use CONCURRENTLY to avoid locking the table:
CREATE INDEX CONCURRENTLY receipts_created_at_idx
  ON public.receipts (created_at DESC);
```

## The bug (reference)

**Root cause:** No index on `receipts.created_at`. The query `WHERE created_at > NOW() - INTERVAL '7 days' ORDER BY created_at DESC LIMIT 25` requires a full table scan to find matching rows, then a sort step to order them.

**Broken state EXPLAIN shape:**
```
Seq Scan on receipts → Sort → Limit
```

**Fix:**
```sql
CREATE INDEX receipts_created_at_idx
  ON public.receipts (created_at DESC);

ANALYZE public.receipts;
```

**Fixed state EXPLAIN shape:**
```
Index Scan using receipts_created_at_idx → Limit
```
